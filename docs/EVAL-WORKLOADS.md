# Evaluation Workload Split

Generated by `scripts/build_eval_workload.py`. Machines read this file via `scripts/run_workload.py --machine <name>`.

Each cell = 1 test case × 1 prompt variant × 1 tier. Each phenotype contributes (# test cases × 9) cells. Split is roughly 75% / 25% by cell count.

## FHIR Resource Types Required for Evaluation

The eval test cases only query a subset of the FHIR resource types Synthea emits. When loading data into the cloud Microsoft FHIR Server via `$import`, upload only the resources below — this skips ~2.6 GB of unused data (ExplanationOfBenefit, DocumentReference, Claim, DiagnosticReport, Immunization, Provenance) and keeps the import to ~422 MB.

Counts derived from `grep '"url":'` across `test-cases/phekb/*.json` (2026-05-04).

### Required (queried directly by test cases)

| Resource | Test-case URL count | NDJSON size | Notes |
|---|---:|---:|---|
| Condition | 233 | 7 MB | Most-queried resource (dx paths) |
| MedicationRequest | 92 | 5 MB | Med paths + cross-indication tests |
| Observation | 49 | 4 MB | Lab/vital threshold tests + multi-coded labs |
| Procedure | 21 | 0.2 MB | 11 procedural specialties |
| Patient | 19 direct + every `_has`/`_revinclude` chain | 65 MB | Also required for stable-ID resolution (Synthea UUID lookup via `Patient.identifier`) |
| Encounter | 1 direct, plus referenced by all clinical resources | 290 MB | Kept for referential integrity |

### Foundation (small, kept for referential integrity)

| Resource | NDJSON size |
|---|---:|
| Organization | 0.7 MB |
| Location | 0.5 MB |
| Practitioner | 19 MB |
| PractitionerRole | 30 MB |

### Skipped (not queried by any test case, ~2.6 GB)

| Resource | NDJSON size |
|---|---:|
| ExplanationOfBenefit | 1.2 GB |
| DocumentReference | 420 MB |
| Claim | 381 MB |
| DiagnosticReport | 332 MB |
| Immunization | 200 MB |
| Provenance | 102 MB |

A handful of test cases reference resource types Synthea doesn't emit (`AdverseEvent`, `MedicationStatement`, `AllergyIntolerance`) — those tests evaluate code/system selection rather than patient-level matching, so no upload is needed.

### Upload target

Azure blob: storage account `jaerwinimport`, container `fhirllmtestresources`. Loaded into Microsoft Health FHIR Server via `$import` in the foundation → Patient → Encounter → clinical-resources order documented in `scripts/build_ndjson_export.py`.
